# 项目主线与交付节奏治理设计

## 目标

让白色凤眼兔桌宠项目在跨会话、跨阶段开发时仍保持同一条主线：不遗忘已确认的方向，不让临时需求无意间打断当前工作，并让用户始终能看清下一步与验收状态。

## 决策

采用“自动推进 + 阶段闸门”的混合节奏：

- 同一已确认阶段内，开发者自动完成测试、实现、验证和小型修复。
- 形象、交互风格、大范围功能、性能取舍、发布与任何会改变主线优先级的事项必须在进入前给用户展示并取得确认。
- 未经用户明确要求，临时想法不改变当前阶段；它们记录到停车场，等待合适的阶段再排期。

## 单一项目驾驶舱

在仓库根目录创建 `PROJECT_STATUS.md`，作为唯一的工作状态入口。它不复述完整规格和实施细节；详细设计仍在 `docs/superpowers/specs/`，具体实施步骤仍在 `docs/superpowers/plans/`。

驾驶舱固定包含：

1. **当前阶段**：阶段名称、目标、开始条件和唯一进行中状态。
2. **主线地图**：桌宠核心 → 互动姿态 → 状态闭环 → 小游戏 → 学堂成长 → 小屋与装扮 → 发布与性能。
3. **已验收成果**：关联提交、自动测试数、用户桌面实测与性能测量结果。
4. **下一步**：最多三项，严格按依赖顺序列出。
5. **停车场**：不影响当前阶段的新想法，包含提出日期、内容与进入主线的条件。
6. **工作规则**：开发开始、完成、阻断与优先级变化时的更新要求。

## 当前主线

当前阶段定为“互动姿态与状态闭环”。目标是在已经可拖动、透明、低资源的桌宠上，增加摸摸、进食、休息的角色姿态和低干扰状态提示；不得引入持续游戏循环或改变既有学堂存档。

后续阶段的推进顺序固定为：

1. 桌宠核心（已完成）
2. 互动姿态与状态闭环（进行中）
3. 小游戏体验闭环
4. 学堂成长扩展
5. 小屋与装扮
6. 发布、性能测量与开机体验

## 执行与验收规则

- 每个阶段都必须走“设计确认 → 测试先行 → 实现 → 自动测试 → 用户 Mac 验收 → 驾驶舱更新”。
- 自动测试失败、关键用户验收失败或超过性能警戒线时，不进入下一阶段。
- 桌宠空闲状态禁止轮询、逐帧循环和常驻实时渲染；每个新增动画或素材必须在设计中说明资源风险。
- 用户提出新需求时，先判断其是否属于当前阶段：直接增强可排入本阶段；涉及视觉/玩法大改时须先提方案；其余写入停车场。
- 每次开始工作先读取 `PROJECT_STATUS.md`；每次形成可验证的阶段成果后更新它；每次停止时必须写明下一步或阻碍。

## 错误与降级

- 如果驾驶舱与详细规格冲突，以用户最新明确确认的规格为准，并立即修正驾驶舱。
- 如果用户明确要求改变优先级，先记录被暂停阶段、原因和恢复条件，再更新主线地图。
- 如果没有可靠的用户桌面验收数据，明确标为“待用户验收”，不得写成已验证。

## 验证

- 检查驾驶舱只存在一个进行中阶段，下一步不超过三项。
- 检查当前阶段、最近提交与已验收信息能由仓库历史和测试输出追溯。
- 在下一次开发会话开始时，确认可以仅通过阅读驾驶舱恢复主线和待办。
